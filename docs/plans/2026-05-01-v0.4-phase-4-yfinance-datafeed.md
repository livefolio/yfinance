**Goal:** Ship `@livefolio/datafeed-yfinance@0.1.0` from a new repo at `~/Documents/Personal/livefolio-2/yfinance/`. The package implements the SDK's `DataFeed` interface (just `bars`) backed by Yahoo Finance via the `yahoo-finance2` npm library. End state: a `tactical/v0` strategy hydrated by `tactical.fromSpec` and run via `runBacktest` with this adapter as its `DataFeed` produces a deterministic, fixture-replayable allocation history. This is the data-side foundation Phase 5's parity test needs.

**Architecture.** The package exports one class — `YfinanceDataFeed` — implementing `DataFeed.bars(asset, range, freq): AsyncIterable<Bar>`. `bars` resolves an `Asset` to a Yahoo symbol (`asset.symbol`), calls `yahoo-finance2`'s `chart` endpoint with `interval: '1d'`, normalizes the response to v0.4 `Bar[]` (UTC timestamps, OHLCV from Yahoo's *adjusted* close path), filters to `range`, and yields. A range-aware in-memory cache (`Map<symbol, { range, bars }>`) deduplicates fetches within a single backtest. Optional capabilities (`fundamentals`, `events`) are not implemented in this phase — they remain `undefined` on the instance, which the SDK's optional `?` discriminator already handles.

**Yahoo daily-bar timestamp quirk.** Yahoo timestamps daily bars inconsistently and the adapter must handle it explicitly. (Verified empirically against `yahoo-finance2@3.14.0` on 2026-05-02: SPY for the prior 10 sessions returned `13:30:00.000Z` for every bar including the freshly-closed May 1 bar; the canonical time-of-day **tracks US market open and shifts with DST** — `13:30Z` during US daylight-saving time, `14:30Z` during US standard time.)

| Bar state | Raw `t` from Yahoo |
|---|---|
| Historical (any prior session) | Canonical session-open in UTC (`13:30Z` during DST, `14:30Z` during ST) |
| Today, market open | current intraday clock (e.g. `17:42:13Z`) — represents an **in-progress** partial bar with a moving close |
| Today, market closed (transition window, possibly minutes to hours) | `~20:00`/`~21:00Z` (close time) — the day's bar exists but hasn't been canonicalized |
| Today, after Yahoo's normalization | canonical session-open (joins the historical pattern) |

A naive "normalize to UTC midnight" pass would silently fold the in-progress bar into the daily series and bake look-ahead into any backtest with `range.to >= today`. The adapter solves this with a **structural completeness filter**:

> **Rule:** drop the last bar iff its UTC time-of-day (`hh:mm:ss`) differs from the modal UTC time-of-day of the preceding bars. Equivalently: trust Yahoo's own canonicalization. If today's bar matches the prior bars' time-of-day, it's been canonicalized → keep. If it doesn't, it's intraday-or-just-closed → drop.

This rule is **DST-agnostic** — it doesn't hardcode `13:30Z` or `14:30Z`, so a fetch that straddles the March or November DST boundary works without calendar knowledge. It's also **provider-trust-only** — the adapter encodes no US-market-hours assumption beyond "Yahoo's prior bars represent the canonical timestamp shape."

After the filter, remaining timestamps are normalized to UTC midnight to match `USEquityCalendar`'s convention. Once Yahoo canonicalizes today's bar (typically within hours of close per the empirical probe), it reappears in subsequent fetches without code changes.

**Edge case — fewer than 2 bars in the response.** With one bar, "modal time-of-day of the preceding bars" is undefined. Behavior: keep the bar. This is the right default — short backtests pulling a tiny range shouldn't lose data, and the look-ahead risk is bounded (the caller asked for a specific narrow range; if it includes today and the bar is intraday, that's on them). Tests cover the 1-bar case explicitly.

**Edge case — DST boundary inside the fetched range.** Bars before and after the boundary will have different times-of-day. Modal time-of-day of "all preceding bars" still picks the dominant one. The boundary-day bar itself is *not* the last bar (unless the fetch happens to end exactly on the boundary day), so it's not subject to the filter — it's a normal historical bar and passes through. Normalization to UTC midnight collapses both times-of-day to the same value.

A constructor option `includeIncompleteToday?: boolean` (default `false`) lifts the filter for live-trading use cases that explicitly want the partial bar. v0.4 backtests use the default.

**Naming convention.** This repo follows the convention "**repo = vendor, package = interface**" (set during phase-4 design):
- Repo: `yfinance` (vendor), under the implicit `livefolio` GitHub org.
- Package: `@livefolio/datafeed-yfinance` (interface-prefixed).
- Future Alpaca repo: `alpaca/`, shipping `@livefolio/datafeed-alpaca` and `@livefolio/executor-alpaca` from one tree.

This **deviates from the SDK's v0.4 spec** at `sdk/docs/specs/2026-04-29-v0.4-multi-repo-interface-design.md` line 32, which sketched `@livefolio/adapter-yfinance`. The interface-prefixed name is preferred because future vendors (Alpaca, IBKR, Polygon) ship multiple capabilities — a flat "adapter" namespace would lose that signal. The SDK spec should be updated to match in a follow-on edit (out of scope for this plan).

**Tech Stack:** TypeScript (strict, `noUncheckedIndexedAccess: true`), ES modules with extensionless imports, Vitest, tsup, Prettier, ESLint. Matches the SDK repo's stack one-to-one so pre-commit hooks and editor configs port unchanged.

**Out of scope for this phase:**
- **`fundamentals` and `events` capability surfaces.** Optional in the `DataFeed` interface; tactical/v0 doesn't read them. Adding them later is a non-breaking minor version bump (the `?` on the interface protects callers).
- **Frequencies other than `1d`.** Phase 3's tactical dialect is daily-cadence by construction (rebalance frequencies are session-counted: `Daily` / `Weekly` / `Monthly` / `Quarterly` / `Yearly`). `yahoo-finance2` supports `1m`/`5m`/`15m`/`60m` — passing through is trivial when a use case lands. For now, calling `bars` with anything other than `'1d'` throws.
- **Persistent caching.** The `MemoryFeatureCache` in the SDK already caches *features*; this adapter caches *bars* in-process for the duration of a `runBacktest` call. Disk caching is a consumer concern (e.g. wrap with a filesystem layer in `livefolio-app`).
- **Live streaming / quotes.** The `DataFeed` interface is historical-bar-shaped. A separate streaming interface lives in the SDK's future; this adapter ships the historical path only.
- **Symbol resolution / asset registry.** `Asset.symbol` is trusted as the Yahoo ticker. The adapter does not maintain its own ticker map. If a consumer needs `Asset.id ≠ symbol`, they construct `Asset` with the right `symbol` field at the call site.
- **Survivorship-bias correction, point-in-time fundamentals, hand-curated splits.** Yahoo's adjusted-close path bakes splits and dividends into OHLC, which the v0.4 spec § Open Questions accepts as the v0.4 fidelity bar.
- **Rate limiting.** `yahoo-finance2` does its own throttling; we don't add a layer. If parity tests hit limits, add a `pacing?` option in a follow-on minor.
- **Codemod for v0.3 strategies.** That's phase 6 in the SDK repo. Phase 4 only delivers the data feed Phase 5's parity test consumes.

Conventions: kebab-case file names; co-located `*.test.ts`; ES module imports extensionless. Tests use Vitest's `vi.fn()` for mocks — no live network in CI. The `test/fixtures/record.ts` script is a one-shot recorder run manually by a developer; CI never executes it.

**Linkage to the SDK.** During development the SDK is unpublished, so `package.json` points at the local checkout:

```json
"peerDependencies": {
  "@livefolio/sdk": "^0.4.0"
},
"devDependencies": {
  "@livefolio/sdk": "file:../sdk"
}
```

`@livefolio/sdk` is a *peer* dependency in published terms (consumer brings their own SDK), and a *dev* dependency in this checkout (so types resolve at build/test time). When the SDK ships 0.4.0 to npm, the `file:../sdk` line is replaced with `^0.4.0` in `devDependencies` and the package can publish.

**Why peer not regular.** Two adapters (e.g. `datafeed-yfinance` + `datafeed-polygon`) used in the same app must agree on one SDK version — if each declared a regular dependency on `@livefolio/sdk`, npm would happily install two copies and `instanceof Asset` checks would silently break. Peer dependencies force the consumer to pin one SDK and warn on conflicts.

  const tod = (d: Date) => d.getUTCHours() * 3600 + d.getUTCMinutes() * 60 + d.getUTCSeconds();

  // Drop last bar if it doesn't match the modal time-of-day of the preceding bars.
  // Single-bar response is a no-op.
  if (!opts?.includeIncompleteToday && quotes.length >= 2) {
    const last = quotes[quotes.length - 1]!;
    const prior = quotes.slice(0, -1);
    const counts = new Map<number, number>();
    for (const q of prior) counts.set(tod(q.date), (counts.get(tod(q.date)) ?? 0) + 1);

    if (tod(last.date) !== modal) quotes = prior;

## Self-Review

1. **Scope vs. parity goal.** Phase 4's stated job in the v0.4 spec § Effort estimate is "implements DataFeed enough to drive parity tests." Phase 5 will name the parity strategy; until then the integration test in Task 7 stands in as proof-of-fit. If the chosen Phase-5 strategy needs `fundamentals` or weekly bars, those land in a phase-4.1 follow-on — not retroactively in this plan.

2. **No SDK edits.** Every task is in the new repo. The SDK is consumed read-only via the `file:../sdk` link. The deviation note about naming (`@livefolio/datafeed-yfinance` vs the spec's `@livefolio/adapter-yfinance`) is *documented* here but the SDK spec edit is a separate one-line follow-up (out of scope for this plan).

3. **Tests don't touch the network.** Task 6's `record.ts` is the only network code; it runs by hand, never in CI. Every other test injects a fixture-backed `fetcher`. The one risk: `yahoo-finance2` might do its own network on `import` (e.g. version-check telemetry). If it does, Task 3's `vi.mock('yahoo-finance2', ...)` blocks it — verified by running tests with the network disabled (`unshare -n` on Linux, or just disconnect WiFi and run `npm test`).

4. **Cache YAGNI on partial overlap.** Task 4's `set` panics on partial-overlap merges instead of supporting them. The call pattern from Task 5 always pulls `[earliest, latest]` once per symbol per backtest, so partial overlap never arises in practice. If a future capability (multi-strategy backtest sharing a feed across non-contiguous ranges) needs it, the panic is the trigger to add merge logic — better than silent wrong answers.

5. **No SDK contract assumptions beyond the public interface.** The adapter implements only `DataFeed.bars`. It does *not* assume anything about how `runBacktest` calls it, what `FeatureRuntime` does with the bars, or whether the SDK's caching layer rounds `range` boundaries. Task 7 verifies the integration; if the SDK changes its iteration pattern, only Task 7 needs to follow.

6. **Forward compatibility.** The class shape is `YfinanceDataFeed`, not `YfinanceDataFeedV1`. Adding `fundamentals` later is non-breaking (interface optional). Adding intraday freqs is non-breaking (the `freq !== '1d'` throw becomes a real implementation). Adding a `pacing?` constructor option is non-breaking (additive optional). The only thing that would force a major bump is changing `bars`'s shape — which is dictated by the SDK's `Bar` type, not this package.

7. **Phase 1 / 2 / 3 untouched.** Confirmed: every task is in the new `yfinance/` repo. Zero edits in `sdk/`. The full SDK test suite (451/451) remains green by construction.

8. **Yahoo timestamp quirk handled at the boundary, DST-agnostically.** The completeness filter (Task 3) and the `includeIncompleteToday` opt-in (Task 5) live in the adapter, where the provider's quirk is. The SDK never sees a non-canonical bar; the runtime trusts `Bar.t === UTC midnight`. The filter rule is **structural** ("last bar's time-of-day must match the modal time-of-day of preceding bars") rather than positional ("must equal `14:30Z`"), so it survives DST boundaries (canonical time shifts `13:30Z` ↔ `14:30Z` in March/November), survives non-US exchanges if added later (canonical time would just be different), and survives any future Yahoo canonicalization shift as long as Yahoo applies the new shape uniformly to historical bars. **Verified empirically** against live `yahoo-finance2@3.14.0` on 2026-05-02 (US DST): 10-day SPY fetch returned `13:30:00.000Z` for every bar including the freshly-closed today bar, confirming the structural pattern. **Remaining edge case:** a fetch of length 1 has no modal reference and the filter is a no-op — documented and tested. The risk envelope is bounded (single-bar narrow ranges that include today).
